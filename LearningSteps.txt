Git File States
```````````````
There are 4 main states:
    1. Untracked -> New file not tracked by Git
    2. Modified -> File is changed but not staged
    3. Staged -> File is ready to commit
    4. Unmodified -> File is already committed and unchanged

Flow Diagram:

    untracked  ──add──>  staged  ──commit──>  unmodified
         ↑               |
         |               ↓
       modified <─────────

--------------------------------------------------------Git Branch--------------------------------------------------------

Branch  -> A branch is a separate line of development in a project. It allows different teams or

Delete the branch
````````````````` 
    git branch -d branchName -> Deletes an existing branch from the local repository.
        Git prevents deletion if the branch contains unmerged changes.
    git branch -D branchName -> Force deletes an existing branch from the local repository.
    Note:
        Before deleting a branch, switch to another branch because Git does not allow deletion 
        of the branch you are currently on.
    Example:
        git checkout main
        git branch -d feature1
        git push origin --delete branchName -> Deletes the branch from the remote repository.
        git fetch --prune
        or
        git remote prune origin -> Removes stale references to deleted remote branches from 
                                   the local repository.
        git remote remove origin -> Removes the remote repository connection (origin) from the 
                                   local repository. It does not delete any branches from GitHub.

-----------------------------------------------Merging branch/Code-----------------------------------------------

We can merge the branches using two ways
1. Using Git
    A branch can be merged into another branch using Git commands from the terminal.
    Steps:
    1. git checkout main -> Switch to the branch where you want to merge the changes(target branch) 
    2. git pull origin main -> Pull the latest changes from the remote repository
    3. git merge <branch-name> -> Merge the source branch into the current branch
        Note: git merge <branch-name>
                merges the specified branch (branch-name) into the branch you are currently on. 
                In other words, it combines the changes from branch-name with the current branch.
    4. If there are no merge conflicts, Git will automatically merge the changes.
    5. git add . -> If a merge conflict occurs, resolve the conflicts in the affected files and 
       stage the resolved files
    6. git commit -m "Merged feature-login into main" -> Complete the merge by creating a merge 
                                                        commit (if required)
    7. git push origin main -> Push the updated branch to the remote repository

After these steps, the changes from the source branch will be successfully merged into the 
target branch.                                                    


2. Using GitHub (PR)
    Pull Request (PR) -> Pull request lets you tell about changes you have pushed to a branch
                        in a repository on GitHub 

    When a new branch is created, the changes can be merged into the `main` branch through GitHub
    using a Pull Request (PR).

    Steps to create and merge a Pull Request:

    1. Go to the specific repository on GitHub.
    2. Open the **Branches** section and select the newly created branch.
    3. Click **Contribute** and then select **Open Pull Request** (or **Compare & Pull Request**).
    4. Add a meaningful title, description, and any relevant comments explaining the changes.
    5. Submit the Pull Request for review.
    6. A senior developer, team lead, or manager reviews the Pull Request.
    7. Based on the review, they may:
        * **Approve and merge** the changes into the `main` branch, or
        * **Request changes** and provide comments for improvement.
    8. After the requested changes are made and approved, the Pull Request can be merged into 
        the `main` branch.

Merge Conflict
``````````````
A merge conflict occurs when Git cannot automatically merge changes because the same part of 
a file has been modified differently in two branches. Git requires a developer to manually 
review and resolve the conflicting changes before the branches can be merged.

For example, If Developer A changes line 10 in the main branch and Developer B changes the 
same line differently in a feature branch, Git will create a merge conflict when attempting 
to merge the branches. The developer must decide which change to keep or how to combine them.

---------------------------------HEAD, Git Log, Reset & Restore---------------------------------
HEAD 
````
HEAD = current position in Git, it shows where you are currently working
Normal state: HEAD → main → latest commit
Detached HEAD state: HEAD → commit (not attached to any branch)
Happens when you checkout a commit: git checkout <commit-hash>

Git Log
```````
git log or git log --oneline -> commit history with hash values

What is commit hash?
-> Each commit has a unique ID (hash) 
-> We use this hash to :
    - Reset to a specific point move 
    - HEAD backward

Git Reset & Restore (Stage-wise Flow)
`````````````````````````````````````
1. Working Directory (Unstaged)

    You modify a file, but it is not yet added to Git staging.
    Restore from Unstaged → undo changes
    git restore <file> -> Removes changes in working directory, file goes back to last committed 
                        version restored to last commit.

2. Staging Area (Staged)

    File is added using  git add .
    Restore from Staged → Unstaged
    git restore --staged <file> -> Removes file from staging area Keeps changes in working 
                                directory
    staged → unstaged

3. Commit (Committed State)

    Changes are saved in Git history
    Commit: A → B → C (HEAD)
    Reset from Commit → move HEAD backward
    git reset -> Used to undo commits

    Step 1: Find commit using log
    ```````
    git log --oneline
    Example:
        a1b2c3d Commit C
        b2c3d4e Commit B
        c3d4e5f Commit A
    We use this hash in reset

    Reset types (Commit level undo) :
    ``````````````````````````````````
    1. Soft Reset (keep staged)
        git reset --soft <hash>
        Commit → Staged
    Undo commit but keep changes ready

    2. Mixed Reset (default)
        git reset <hash>
        Commit → Unstaged
    Undo commit and unstage changes
    
    3. Hard Reset
        git reset --hard <hash>
        Commit → Deleted completely
    Deletes all changes permanently

    * Single Reset
    Before:
    A → B → C (HEAD)
    Command:
    git reset --hard <B_hash>
    After:
    A → B (HEAD)
    * Multiple Reset
    Before:
    A → B → C → D → E (HEAD)
    Command:
    git reset --hard <B_hash>
    After:
    A → B (HEAD)
    All commits after B are removed from history

    RESET vs RESTORE
    ````````````````
    Feature	        reset	        restore
    ```````````````````````````````````````````
    Level	        Commit level	File level
    Uses	        git log hash	file name
    Affects HEAD	Yes	            No
    Risk	        High	        Low

--------------------------------------Git Fork --------------------------------------------

Fork -> Copy of a repository on GitHub
        It creates your own version of someone else’s repository
        You can freely make changes without affecting the original project

Fork Flow (Real-world idea)
    Original Repo (Owner)
            ↓ Fork
    Your GitHub Copy (Forked Repo)
        1. Original Repository (Source)
        This is the main project on GitHub
        You can view and clone
        You usually don’t have write access
        Owner Repo → main project

        2. Forking the Repository
        When you click Fork on GitHub
        What happens -> GitHub creates a copy of the repo in your account
                        You now have full control of your copy
                        Original Repo → Your Forked Repo
        
        3. Your Forked Repository
        This is now your independent copy
        You can -> modify code/create branches/commit changes
        Your Repo (Forked) → full control
        
        4. Clone to Local (Optional step)
        git clone <your-fork-url>
        GitHub Fork → Local Machine
        Now you work locally like normal Git project:
        modify files -> stage changes -> commit changes
        
        5. Push Changes to Your Fork
        git push origin main -> Local to Your Forked Repo

        6. Pull Request (PR)
        After changes are done:
        You send changes back to original repo using PR

Your Fork → Pull Request → Original Repo

Full Fork Workflow
``````````````````
        Original Repo
              ↓ Fork
        Your GitHub Repo
              ↓ Clone
        Local Machine
              ↓ Changes + Commit
        Your Forked Repo
              ↓ Pull Request
        Original Repo (Owner)

Why Fork is used ?
-> You don’t have permission to directly modify main repo
-> You can experiment safely
-> Used in open-source projects
